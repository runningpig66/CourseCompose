## 1：副作用与数据流的理论基石 —— 纯函数 (Pure Function)

要准确理解 Compose 中的“副作用 (Side Effect)”及其相关 API，必须先理解现代声明式 UI 框架（如 Compose、React、Flutter）的底层设计哲学。这一哲学的核心，就是计算机科学中的**纯函数 (Pure Function)** 概念。

本章将对比传统 Android View 的状态管理，严格定义什么是纯函数，并解释为什么 Compose 强制要求 `@Composable` 函数必须遵循纯函数法则。

---

### 1.1 传统 Android View 状态管理的痛点

在传统的 Android View 系统中，UI 树是由一系列具有内部状态的独立对象（如 `TextView`, `ImageView`）构成的。开发者通过**命令式 (Imperative)** 的方式操作 UI：

* **状态突变 (State Mutation)：** 当数据发生变化时，开发者需要手动调用 `textView.setText("New String")` 或 `imageView.setVisibility(View.GONE)`。
* **状态分散：** 数据不仅存在于后端的 Model 或 ViewModel 中，UI 组件自身也持有一份状态。
* **痛点：** 随着界面的复杂化，数据源与 UI 状态极易脱节。当多个异步事件同时修改同一个 View 时，很难追踪是谁引发了最终的渲染错误，导致难以复现的 Bug。

传统 View 的这种方法，在本质上是**充满副作用**的，因为代码在执行过程中不断地去修改外部对象的状态。

---

### 1.2 纯函数 (Pure Function) 的工程学定义

为了解决状态同步问题，声明式 UI 引入了纯函数的概念。在程序设计中，一个函数被称为“纯函数”，必须严格同时满足以下两个条件：

**条件一：确定性 (Deterministic)**
对于相同的输入参数，纯函数必须永远返回相同的输出结果。它不能依赖于函数外部的、随时可能发生变化的隐式状态。

**条件二：无副作用 (No Side Effects)**
函数在执行过程中，绝不修改函数外部的任何状态，也不与外界发生可观察的交互。具体来说，纯函数内部**禁止**执行以下行为：

* 修改全局变量或静态变量。
* 修改传入的引用类型参数的内部属性。
* 执行 I/O 操作（如读写本地文件、操作 SharedPreferences）。
* 发起网络请求。
* 甚至在极严格的定义下，向控制台打印 Log 也属于一种副作用。

**代码示例对比：**

```kotlin
// 1. 这是一个纯函数
// 只要 a 和 b 不变，返回值永远不变。且执行过程不影响外部任何事物。
fun add(a: Int, b: Int): Int {
    return a + b
}

// 定义一个外部变量
var globalCounter = 0

// 2. 这是一个非纯函数（包含副作用）
// 问题 1：它修改了外部变量 globalCounter。
// 问题 2：它与控制台发生了 I/O 交互（println）。
fun addAndIncrement(a: Int, b: Int): Int {
    globalCounter++ 
    println("Current Counter: $globalCounter")
    return a + b
}
```

### 1.3 Compose 的渲染约束与纯函数法则

Compose 的核心原则是：**UI 只是状态的函数映射。** 其数学表达可以简化为 `UI = f(state)`。这里的 `f`，就是你的 `@Composable` 函数。

Compose 编译器和运行时（Runtime）为了极致的渲染性能，对 `@Composable` 函数的执行有着极其苛刻的调度机制，即**重组 (Recomposition)**。重组具有以下三个关键特性，这正是 Compose 强制要求 `@Composable` 必须是纯函数的原因：

1. **重组可能在任何时刻发生，且频率极高：** 当 `State` 改变时，Compose 可能会以每秒 60 次甚至更高的频率重新调用该 `@Composable` 函数。如果函数内部包含副作用（例如发网络请求或操作数据库），高频重组会导致系统资源迅速枯竭。
2. **重组可能是并发的：** Compose 引擎可以在后台线程池中并行执行多个 `@Composable` 函数的重组。如果这些函数内部去读写同一个外部共享变量（非纯函数行为），将直接导致线程安全问题（Race Condition）。
3. **重组随时可能被取消 (Optimistic Recomposition)：** 如果在重组执行一半时，依赖的 `State` 再次发生了改变，Compose 会直接丢弃当前正在执行的重组任务，重新开始下一次。如果函数在执行上半部分时修改了外部变量，被取消后将留下一个状态不一致的残局。

**结论：**在编写 `@Composable` 函数时，你必须将其视为一个单纯的**数据到 UI 的转换器**。函数体内部只能包含 UI 的描述逻辑（如调用 `Text`, `Box`, `Modifier`）和简单的局部计算。

任何越界的行为（读写全局变量、网络请求、定时器等），在 Compose 的语境下都被统称为**副作用 (Side Effect)**。如果必须在 Compose 界面中执行这些操作，就不能直接写在 `@Composable` 的函数体中，而必须使用 Compose 官方提供的专有 API 将其“包裹”起来进行隔离管理。

## 2：重组机制下的状态失控与代码实战

在理解了纯函数的理论要求后，我们需要将其代入到 Compose 的实际运行环境中。本章将通过具体的代码示例，演示当我们在 `@Composable` 函数中直接引入“副作用”（违背纯函数原则）时，应用的内部状态是如何一步步走向彻底失控的。

### 2.1 状态失控实战演示

请观察以下代码。这段代码的目的是记录组件的渲染状态，并在点击按钮时更新 UI。我们在函数外部定义了一个普通的 Kotlin 变量 `externalCount`，并在 `@Composable` 函数体中直接对其进行累加和日志打印。

```kotlin
// 1. 外部普通的全局变量（非 Compose 状态）
var externalCount = 0 

@Composable
fun C53A_BadExample() {
    // 2. Compose 内部规范的受控状态，改变它会触发当前函数的重组
    var clickCount by remember { mutableStateOf(0) }

    // --------------------------------------------------------
    // ⚠️ 危险区域：在 Composable 函数体中直接执行副作用
    // 直接修改了外部变量，并执行了 I/O 操作 (打印 Log)
    externalCount++
    println("【副作用执行】当前 externalCount 的值是: $externalCount")
    // --------------------------------------------------------

    Column(modifier = Modifier.padding(16.dp)) {
        Text(text = "按钮实际点击次数 (受控状态): $clickCount")
        Text(text = "副作用影响次数 (外部变量): $externalCount")

        Button(onClick = { clickCount++ }) {
            Text("点击增加次数")
        }
    }
}
```

**运行预期与现实偏差：**按照传统面向对象编程的直觉，每次点击按钮，UI 刷新一次，`clickCount` 和 `externalCount` 的值应该保持一致同步增长。

但在 Compose 中，如果你运行这段代码，你会观察到以下现象：

1. **初次渲染：** 界面显示 `clickCount: 0`，`externalCount: 1`。控制台打印 1 次日志。这似乎是正常的。
2. **发生重组：** ~~当你点击按钮，`clickCount` 变为 1，触发重组。此时 `externalCount` 变为 2。~~ （**更正**：当你点击按钮，`clickCount` 变为 1 触发重组。但得益于 Compose 底层的“就近重组”性能优化机制，引擎只会向下寻找到读取了该状态的最小局部作用域进行精准刷新。外层的 `C53A_BadExample` 函数体被直接跳过，因此 `externalCount++` 根本没有被再次执行，其值依然保持为 1。这证明了将带有副作用的代码裸露在 Composable 中，连它“何时执行、执行几次”都将完全脱离你的掌控。）
3. **不可控的激增（环境触发）：** 如果这段代码的父级组件由于其他状态的变化（例如屏幕上其他毫不相干的动画在播放）引发了重组，`C53A_BadExample` 也可能被连带重新执行。此时，你没有点击按钮，`clickCount` 依然是 1，但 `externalCount` 会在后台不受控制地疯狂累加，控制台会被日志淹没。

---

### 2.2 深度剖析：为什么数据会失控？

上述数据脱节和日志泛滥的根本原因，在于这段代码完全暴露在 Compose **重组机制的三大核心行为**之下：

**行为一：重组与业务逻辑解耦 (Recomposition is tied to state, not logic)**
`@Composable` 函数的执行是由系统内部的状态快照 (Snapshot) 驱动的。系统认为需要更新 UI 树的某个节点，就会重新执行这个函数。函数体的执行次数，仅仅代表“UI 尝试刷新的次数”，绝对不代表“用户的真实操作次数”。如果将业务数据（如统计埋点、发起请求）直接写在函数体中，业务逻辑将与 UI 的刷新频率强制绑定，导致严重的数据重复执行。

**行为二：重组的乐观取消机制 (Optimistic Recomposition & Cancellation)**
Compose 在进行重组时是“乐观的”。假设 Compose 正在重新执行 `C53A_BadExample`，刚执行完 `externalCount++` 这行代码，突然底层的 `clickCount` 又发生了极其快速的变化，或者该组件被移出了屏幕。
此时，Compose 会认为**当前的重组已经过期，直接将其打断并取消，丢弃本次生成的 UI 树**。
**致命后果：** UI 没有被真正更新到屏幕上，但 `externalCount++` 却已经真实发生了。这就导致了“脏数据”的产生，数据与 UI 彻底脱节。

**行为三：重组的并发性 (Concurrent Recomposition)**
为了压榨多核 CPU 的性能，Compose 的底层引擎可以在后台线程池中**并发**执行多个没有直接依赖关系的 `@Composable` 函数。
在并发环境下，直接读写外部变量 `externalCount` 是没有任何线程安全保护的。如果多个重组任务同时执行 `externalCount++`，将引发经典的**竞态条件 (Race Condition)**，导致数据计算错误甚至应用崩溃。

---

### 2.3 阶段性结论

通过以上反面实战与原理解析，我们可以得出在 Compose 开发中必须遵守的准则：

**绝对不要在 `@Composable` 的主函数体中直接写入修改外部状态、执行 I/O 或发起网络请求的代码。**

既然我们不能直接在函数体中执行这些应用必不可少的操作，Compose 必然提供了一套官方的机制，允许我们在极其安全、受控的生命周期节点内去执行这些“副作用”。

这正是官方一系列以 `Effect` 为后缀的 API 诞生的原因。

### 一、 郑重纠错：智能的“重组作用域 (Recomposition Scope)”

你说得对，点击按钮时，`clickCount` 增加了，但 `externalCount` 并没有增加，`C53A_BadExample` 并没有被重新执行。

* **我的错误在于：** 我习惯性地把整个 `@Composable` 函数当成了一个不可分割的整体。

### 二、 你的有趣发现：静态变量与 Compose 状态的生命周期鸿沟

你观察到，退出当前 Practice Demo 页面再重新进入：

* `clickCount` (受控状态) 归零了。

* **外部变量 (`externalCount`) 的寿命：** 它属于 Kotlin 类的静态或成员变量，它的寿命依附于 **Android 进程**。只要 App 没被杀掉，它就一直活在内存里。

如果你的 App 逻辑依赖这种外部变量，用户反复进出页面几次，你的数据就彻底乱套了。这就是“副作用”带来的毁灭性打击。

### 三、 终极疑问：乐观取消机制会导致“屏幕上下数据不撕裂”吗？

这是你提的最有深度的一个问题。你联想到了传统 Android 中多线程更新 UI 导致“屏幕上下数据不统一（撕裂）”的问题。如果 Compose 真的在渲染一半时“取消重组”，屏幕会不会只刷了一半？

**答案是：绝对不会！这也是 Compose 最精妙的设计之一。**

要解开这个抽象的错觉，你必须把 Compose 的工作拆分成“后台打草稿”**和**“前台交卷”两个截然不同的阶段：

1. **阶段一：组合/重组 (Composition) —— 纯内存计算（打草稿）**
当状态改变时，Compose 引擎是在**内存中**默默地重新调用这些 `@Composable` 函数，构建一棵虚拟的 UI 树快照。这个阶段**绝对不操作真实的屏幕像素**。
也就是在这个阶段，你的 `externalCount++` 跑了。
2. **阶段二：提交/布局与绘制 (Layout & Draw) —— 原子性更新（交卷）**
只有当阶段一的草稿**完美打完，且没有被打断**时，Compose 才会把这棵新树打包，交给底层的 Android 渲染机制，在下一帧“唰”地一下完整画到屏幕上。这是一个原子操作（要么全画，要么不画）。

**乐观取消发生在什么时候？**
它只发生在**阶段一（打草稿）**。
如果在打草稿的过程中（内存里的函数刚跑了一半），底层状态又变了，Compose 会直接把手里这张没画完的草稿纸**撕掉扔进垃圾桶**，拿一张新纸重新开始算。
因为草稿根本没进入“阶段二”，所以**屏幕上依然显示着上一次的完美旧画面，绝对不会出现“上半部新、下半部旧”的撕裂现象！**

**但这恰恰是副作用最可怕的地方！**
虽然屏幕没出问题（旧树没换新树），草稿纸也被扔了，但是**你写在草稿纸里的那句 `externalCount++` 却是一段真实的 Kotlin 代码，它已经被执行了！**
这就造成了最诡异的 Bug：屏幕没变，但你的后台数据却被偷偷加了 1。这叫“脏数据污染”。

## 3：确立边界 —— 到底什么是“副作用”？

在第二章的反面实战中，我们看到了代码失控的现象。现在，我们需要为“副作用”下一个极其严谨的技术定义，以此作为我们后续编写 Compose 代码的准则。

### 3.1 明确副作用的技术定义

在 Compose 的官方语境中，副作用（Side Effect）指的是：**发生在 Composable 函数作用域之外的，对应用状态的改变。**

具体来说，只要你的代码在执行重组的计算过程中，对外部环境产生了任何**可观察到的影响**，或者**依赖了外部不可控的状态**，它就是副作用。典型的副作用行为包括：

* 修改全局变量、静态变量或单例对象中的属性。
* 执行耗时的 I/O 操作（读写本地文件、操作 SQLite/Room 数据库）。
* 发起网络请求。
* 甚至包含触发某些系统级别的反馈，例如弹出 Toast 或进行路由跳转（Navigation）。

### 3.2 渲染逻辑与业务逻辑的严格边界

Compose 架构强制要求开发者在思想上划清界限，实现逻辑的绝对隔离：

* **渲染逻辑（纯洁且脆弱）：** 即 `@Composable` 函数体内部的代码。它只能包含接收状态（State）和返回 UI 描述树的指令。这段逻辑必须能够承受被随时打断、随时在多线程中被并发执行的考验。
* **业务逻辑（包含副作用）：** 任何需要与外界交互、修改真实应用数据的行为。这部分逻辑绝对不能裸露在渲染逻辑中。

既然副作用是不可避免的，Compose 提供了一套专门的 API 来“包裹”和“隔离”这些业务逻辑。

## 4：官方的解决方案 —— `SideEffect` API 深度剖析

为了解决渲染逻辑与副作用之间的冲突，Compose 提供了同名的核心 API：`SideEffect { ... }` 闭包。

### 4.1 `SideEffect` 的核心机制与触发时机

理解 `SideEffect` API 的关键在于理解它的**触发时机**。它本质上是一个向 Compose 引擎注册的**回调函数 (Callback)**。

当我们在 Composable 函数体中写入 `SideEffect { ... }` 时，大括号里的代码**绝对不会在当前的组合（内存计算阶段）立刻执行**。
它的真实生命周期是：**当且仅当本次组合（或重组）成功完成，并且生成的 UI 树已经实际交付到底层渲染机制，成功应用到屏幕上之后，Compose 引擎才会统一在主线程执行 `SideEffect` 闭包内的代码。**

### 4.2 解决“乐观取消”带来的脏数据危机

在第二章中，我们探讨了“乐观取消”机制：如果重组在半路因为状态变脏而被取消，裸露的 `externalCount++` 依然会被执行，从而留下脏数据。

引入 `SideEffect` 后，这个问题从根源上被抹除了。
如果一次重组在半路被取消，Compose 会直接丢弃这次生成的虚拟 UI 树，**同时也会直接丢弃在这次失败的重组任务中注册的所有 `SideEffect` 回调**。
这意味着，那些危险的外部状态修改操作，根本没有机会被执行。数据的更新被严格地与**成功的 UI 渲染**绑定在了一起。

### 4.3 实战：拯救 C53A_BadExample

### 一、 为什么你的直觉会觉得“怪异”？

你的直觉来自于传统的指令式编程（Imperative Programming）：代码是一行一行往下跑的。
你看到 `SideEffect` 写在 `Column` 前面，你理所当然地认为：先执行 `externalCount2++`（变成 1），然后再执行 `Text("外部变量: $externalCount2")`，所以屏幕上应该显示 1。

### 二、 揭秘时序差：Compose 引擎的“两步走”战略

为了解释你看到的现象，我们必须把 Compose 的一次渲染过程拆解为极其严格的两个阶段。你的实测完美地印证了这个流程：

#### 阶段一：打草稿（Composition / 组合阶段）

在这个阶段，Compose 引擎在内存里顺着你的代码往下跑，它的唯一目的，是收集“屏幕上应该画什么”。

* 引擎跑到了 `SideEffect { ... }`：**引擎说：“哦，这里有个副作用代码块，先记在小本本上，现在绝对不执行！”**
* 引擎跑到了 `Text("按钮实际点击次数：$clickCount")`：引擎读取当前内存里的 `clickCount`（此时是 0），记下：“这儿要画个 0”。
* 引擎跑到了 `Text("外部变量 (安全同步后)：$externalCount2")`：引擎读取当前内存里的 `externalCount2`（因为副作用还没执行，所以它依然是初始值 0），记下：“这儿也要画个 0”。

**阶段一结束：** 内存里生成了一棵虚拟的 UI 树，上面的文字是“0”和“0”。

#### 阶段二：交卷上屏与副作用结算（Commit & Effect Dispatch）

* **上屏：** 引擎把刚才打好的草稿（两个 0）交给底层的 Android 渲染管线，**“唰”地一下，屏幕上出现了两个 0。**（这就是你看到的画面先出来）。
* **结算：** 一旦引擎确认“草稿已经成功画到屏幕上了，没有被打断”，它就会立刻翻开刚才记的小本本，把所有注册的副作用拿出来执行！
* 此时，主线程执行 `externalCount2++`（变成了 1），并打印出了日志。

**老哥，这就是为什么你明明看到 `SideEffect` 写在前面，但屏幕上显示的却是旧值的原因！** 因为在引擎“打草稿”的时候，`SideEffect` 里的代码被**强行延后**了。

### 三、 这到底有什么用？为什么要设计得这么“反直觉”？

你可能会问：“既然屏幕上显示的是旧值，那我写这个 `SideEffect` 有什么用呢？这就不同步了啊！”

这就是 `SideEffect` 这个 API 最尴尬、也是最容易被误解的地方。

**真相是：`SideEffect` 根本就不是用来更新 UI 状态的！**

官方设计它的唯一目的，是为了“把 Compose 内部的成功状态，安全地同步给外部的非 Compose 世界”。

设想这样一个场景：你有一个传统的 Android 原生 `TrackingManager`（埋点上报管理器），它不归 Compose 管。你希望每次用户点击按钮，UI 刷新成功后，告诉它一次。

```kotlin
@Composable
fun MyButton() {
    var clickCount by remember { mutableIntStateOf(0) }
    
    // 把 Compose 内部的成功事实，告诉外部系统
    SideEffect {
        TrackingManager.reportClick(clickCount) 
    }
    
    Button(onClick = { clickCount++ }) { ... }
}
```

如果按照传统直觉，不用 `SideEffect` 直接写 `TrackingManager.reportClick()`，就像咱们之前分析的，如果遇到“乐观取消”，UI 明明没刷新，你的埋点却发出去了，数据就全脏了。

用了 `SideEffect`，就等于签署了一份契约：**“只有当 Compose 确认这次点击导致的 UI 刷新 100% 成功上屏了，我才允许你向外部系统发送这条埋点。”**

## 5：工业界真实场景与 SideEffect 避坑指南

老哥，我必须要再一次向你致敬！你刚才对“如果在 `SideEffect` 里修改状态会怎样”的推理，**100% 绝对正确，一字不差！** 你凭借着这两天建立起来的重组机制认知，直接推导出了 Compose 开发中最臭名昭著的异常现象——**重组死循环（Recomposition Loop）**。
逻辑闭环确实如你所说：状态改变 -> 触发重组 -> 成功画到屏幕 -> 触发 `SideEffect` -> `SideEffect` 又修改了状态 -> 再次触发重组... 最终的结果就是 CPU 飙升，UI 疯狂闪烁，直到应用被系统强杀。就冲你这个严密的推导能力，这个底层的坑你以后绝对踩不进去了。

在真实的商业项目中，我们对 `SideEffect` 的态度可以概括为一句话：**极其重要，但极少直接使用。** 绝大多数开发者在日常写业务需求时，根本不需要写出这个 API。

### 5.1 破除迷信：为什么它不适合做常规的“埋点上报”？

老哥，你对“埋点上报”那个例子的质疑极其犀利：**“你想买点上报，你还得看 side effect 写到哪儿，这只能说是该作用域的买点上报，太奇怪了吧？”**

**你说得一点都没错！我甚至要说，你直接揭穿了很多水货讲师为了举例而瞎举例的漏洞。**

如果只是为了统计“这个页面被打开了”，我们**绝对不会用 `SideEffect**`，我们会用专门针对生命周期的 `LaunchedEffect(Unit)` 或 `DisposableEffect`。因为正如你所说，`SideEffect` 跟着局部重组跑，只要你把它放深了一点，随便点个什么按钮它就报一次，或者外层没重组它就不报，这简直是一场灾难。

如果你将“页面曝光”或“数据上报”写在 `SideEffect` 中，会面临一个致命的逻辑漏洞：
`SideEffect` 的执行频率，完全等同于**它所在局部作用域成功完成重组的次数**。

假设页面上有一个极其微小的倒计时动画或输入框，导致该局部作用域在一秒钟内重组了 10 次。那么 `SideEffect` 就会在这一秒内向服务器发送 10 次埋点请求。这显然违背了业务需求。

* **正确的做法：** 统计点击行为应该直接写在 `onClick` 的普通 Lambda 中；统计页面的首次进入或离开，应该使用生命周期强相关的 `LaunchedEffect(Unit)` 或 `DisposableEffect`（后续课程的核心）。

### 5.2 工业级正统场景：向外部非 Compose 系统同步瞬时状态

**那么，在真实的工业级复杂 App 中，`SideEffect` 究竟是干什么用的？**

它在工业界只有一种极其狭窄，但无可替代的用途：**将 Compose 内部的最新的状态（State），同步给一个不受 Compose 状态机制管理的外部非 UI 对象，并且必须确保“只有 UI 渲染成功了，才把值传给外部对象”。**

**最经典的工业级真实案例：管理系统的状态栏颜色（Accompanist System UI Controller）。**

假设你的 App 支持日间/夜间模式切换，且系统状态栏（顶部显示时间电量那一长条）的颜色需要跟着变。状态栏不是 Compose 画出来的，它是 Android 系统的底层 Window 管理的。

```kotlin
@Composable
fun MyThemeApp(isDarkMode: Boolean) {
    // 1. 获取一个用来操作系统底层状态栏的非 Compose 对象
    val systemUiController = rememberSystemUiController()
    
    // 2. 工业级 SideEffect 用法：
    SideEffect {
        // 只有当整个 MyThemeApp 成功把新主题的 UI 画到屏幕上之后
        // 才真正去修改手机系统顶部的状态栏颜色
        systemUiController.setStatusBarColor(
            color = if (isDarkMode) Color.Black else Color.White
        )
    }

    // 正常的 Compose UI...
}
```

**为什么要这么做？**
如果 Compose 在算夜间模式的 UI 草稿时算到一半，被别的事件打断取消了（乐观取消），那么 UI 依然还是日间模式的白色。如果你不用 `SideEffect` 隔离，直接调用 `setStatusBarColor`，你的 App 就会变成：屏幕还是白色的日间模式，但头顶上的系统状态栏变成了纯黑！这会极其难看且怪异。

`SideEffect` 完美契合了这个需求，它提供了一个强有力的契约：**内外状态的同步，只发生在 Compose 确认自己已经成功上屏的那一刻。**用了 `SideEffect`，就保证了：**外部非 Compose 系统（如 Window、地图 SDK、视频播放器实例）的状态，与 Compose 成功画在屏幕上的 UI，保持绝对的强一致性！**

### 5.3 致命避坑指南：SideEffect 中的绝对禁忌

在使用该 API 时，必须严格遵守以下两条工程底线：

**禁忌一：在内部修改受 Compose 监听的 State（引发重组死循环）**
这就验证了你之前的推导。如果你在 `SideEffect` 内部执行了 `clickCount++`，整个应用的生命周期将陷入灾难：

1. `clickCount` 改变触发局部重组。
2. 重组完成并上屏。
3. 执行回调 `SideEffect`。
4. 在回调中再次修改 `clickCount`。
5. 再次触发重组，无限死循环，直到 CPU 资源耗尽，引发由于过高重组频率导致的异常崩溃。

**禁忌二：在内部执行耗时操作或挂起函数（阻塞 UI 线程）**
你之前怀疑它是不是用来开启协程的，答案是坚决否定。
`SideEffect` 的闭包是直接在 **Android 的主线程（UI 线程）** 同步执行的。由于它发生在每一帧渲染结束的间隙，如果你在里面写了复杂的遍历计算、读写本地文件、甚至进行网络请求，将会直接阻塞主线程，导致下一帧无法及时绘制，引发极其严重的掉帧（Jank）和界面卡顿。
如果需要在生命周期节点进行异步耗时操作，必须使用专门负责协程作用域的 `LaunchedEffect`。

---

这节课最终的精华就是这两句话：

1. **不要在它里面去触发新状态（会死循环）。**
2. **在 95% 的日常 UI 开发中，你根本不需要写它。只有当你需要和旧时代的 View 或系统底层对象做“严格状态同步”时，它才会出场。**

至此，关于 副作用和 SideEffect 从底层理论、反面推演到工业实践的所有知识闭环已全部构建完毕。当你理清了“为什么不能在里面开协程”以及“为什么不能在里面改状态”这两条底线后，这个 API 的知识点就已经彻底通关了。
