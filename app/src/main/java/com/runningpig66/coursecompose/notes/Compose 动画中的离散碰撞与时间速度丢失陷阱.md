## 剖析：Compose 动画中的离散碰撞与“时间/速度丢失”陷阱

### 背景

在使用 `Animatable` 驱动动画时，如果采用“碰撞后截断坐标 (`updateBounds`) -> `while` 循环检测停止状态 -> 开启反向新动画”的离散碰撞检测（Discrete Collision Detection）方案，会在物理表现上产生不易察觉但致命的“黏墙效应”与“动能误差”。本文通过微观的时间轴（1 帧 = 16.6ms）推演，揭示该方案在底层存在的两处物理精度丢失。

### 1. 设定一个精确的实验场景

为了计算绝对清晰，我们假设一个没有摩擦力的匀速模型（这样更好算，最后我们再把摩擦力加进来）：

* **画布环境**：可用滑动宽度 `maxWidth = 1000dp`。
* **小方块**：宽度 `100dp`。所以右侧的**绝对物理墙壁坐标是 `900dp**`。
* **瞬时状态**：在第 `0` 毫秒（此时画面刚渲染完第 N 帧），小方块正向右滑动，当前坐标是 **`x = 820dp`**。
* **当前速度**：**`100dp / 帧`**（也就是每过 16.6 毫秒，它理应往前走 100dp）。

---

### 2. 完美的物理世界（数学折叠法/真实情况）

在真实的物理世界里，从 `0` 毫秒到 `16.6` 毫秒（也就是计算下一帧画面的这个周期内），会发生什么？

* 小方块从 `820dp` 出发，向右走 `80dp`，到达了 `900dp` 的墙壁。
* 这个过程占用了本帧多少时间？占用了本帧的 **80%** 的时间（因为 80/100 = 0.8）。
* **重点来了**：这 16.6 毫秒的帧周期还没有结束！还剩下 20% 的时间。
* 真实的小方块会在这剩下的 20% 时间里，瞬间反弹，向左走剩下的 `20dp`（100 * 20%）。
* 所以，当 `16.6` 毫秒到来，系统把第 N+1 帧画到屏幕上时，**小方块应该画在坐标 `880dp` 的位置。**（900 - 20 = 880）。

**结论**：在完美的物理模型里，你在屏幕上看到的连续两帧是：`820` -> `880`（已经弹回来了）。

---

### 3. 我们代码里的现实（`while` 循环 + `updateBounds`）

现在我们来看看，讲师带咱们写的这段“撞墙再开启新动画”的代码，在同样的 16.6 毫秒里，底层发生了什么见不得人的勾当：

* **0 毫秒（第 N 帧）**：小方块画在 `820dp`。
* **底层的 16.6 毫秒计算中**：动画引擎计算下一帧的位置：`820 + 100 = 920dp`。
* 引擎拿着 `920dp` 去问 `updateBounds`。`updateBounds` 一看，大喊一声：“越界了！墙只有 900dp！”
* 于是，引擎粗暴地把小方块的坐标**强行截断，赋值为 `900dp**`，并立刻向外抛出一个 `BoundReached` 的状态。
* **16.6 毫秒（第 N+1 帧渲染）**：屏幕刷新，**小方块被画在了 `900dp` 的位置，死死贴在墙上！**
* **16.6 毫秒 ~ 33.2 毫秒**：你的 `while` 循环在这个时候被触发了，它拿到状态，急急忙忙地新 `launch` 了一个向左的动画。
* **33.2 毫秒（第 N+2 帧渲染）**：新动画算出了第一帧，小方块向左移动 100dp，**被画在了 `800dp` 的位置**。

### 4. 两者的致命差异（时间与空间的丢失）

对比两者的画面渲染轨迹：

* **真实物理画面**：820 -> 880 (已反弹) -> 780
* **代码循环画面**：820 -> **900 (粘在墙上)** -> 800

这就是“时间丢失/黏墙效应”的本质。在 `while` 循环模型里，小方块在第 N+1 帧的时候，由于 `updateBounds` 的粗暴截断，**它把本该用来反弹的那 20% 的帧时间，白白消耗在了“停在墙上”这件事上。**

如果初始速度非常快（比如一帧能走 500dp），这种吃掉帧内剩余时间的现象，就会在视觉上表现为：**小方块撞墙的那一瞬间，明显停顿/卡住了一下，然后才弹回来。**

---

### 5. 补充第一点：速度与摩擦力的丢失

引入摩擦力衰减后，“动能误差”会随之暴露：

* 在真实物理中，那被吃掉的 20% 的时间里，小方块在往回弹，**摩擦力依然在工作**，速度依然在变慢。
* 但是在 `while` 代码里，引擎在 `900dp` 撞墙的瞬间，就把那个时刻的瞬时速度记录下来了。而代码直接拿着这个旧速度，当做了第 N+2 帧的初始速度！
* **结果**：小方块在这 20% 的时间里，不仅停止了移动（黏墙），还**享受了一个“摩擦力真空期”**（速度没有在这段时间里继续衰减）。

---

### 结论与终极架构方案

通过上述的逐帧推演可以得出，采用 `updateBounds` 配合 `while` 循环重新 `launch` 的做法，本质上属于物理引擎中离散碰撞检测（Discrete Collision Detection）的一种粗糙实现。

**该方案的致命缺陷在于：**
只要物理碰撞不是极其苛刻地完美发生在一帧的绝对结尾（现实中概率极低），系统就必定会吞噬掉该单帧碰撞后的剩余时间，并顺带抹除这段时间内的摩擦力衰减计算。这最终导致了视觉上的“黏墙卡顿”与物理上的“动能误差（反弹初速度偏高，能量不守恒）”。

**终极解决方案：空间折叠法（Triangle Wave Mapping / 画纸切割法）**
为了实现极高精度的无损物理反弹，应彻底废弃底层的碰撞截断。

1. **物理计算层**：取消边界约束，让动画状态机在一条一维的无限直线上，按照真实的衰减曲线持续运行。这保证了时间流逝的一致性和摩擦力计算的绝对连贯（0 时间丢失，0 动能断层）。
2. **UI 渲染层**：仅在最后一步（如 `Modifier.offset`），利用取模运算（Modulo）和数学逻辑，将这个无限延伸的单向坐标“折叠”回可视的物理视口内。

这种“底层无限奔跑，UI 层数学折叠”的架构思想，完美规避了离散碰撞带来的所有副作用，是解决高保真连续反弹动效的最优解。
